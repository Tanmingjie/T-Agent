# T-Agent Recorder：需求背景与范围

> 状态：已确认  
> 归档日期：2026-07-06  
> 本文只记录需求与产品边界，不代表最终技术实现细节。

## 1. 背景

新版本上线初期通常缺少可执行的自动化用例。传统自动化脚本需要由具备编码能力的自动化测试人员编写，建设进度往往比业务版本慢若干个版本，导致首版本主要依赖人工回归。

本需求希望通过录制测试人员在浏览器中的操作，直接生成标准、可见、可复制和可下载的 Python 自动化测试脚本，缩短从手工测试用例到自动化脚本之间的距离。

它解决的是“自动化脚本创建效率”问题，不是建设一个新的自动化执行平台。

## 2. 目标用户

主要用户是测试经理或承担用例设计工作的测试人员：

- 熟悉业务和测试场景；
- 能编写手工测试用例；
- 通常不熟悉或不愿直接编写自动化代码；
- 能根据录制结果完成本地回放验证；
- 可以接受在页面元素发生变化后，由人员维护生成的脚本。

## 3. 核心场景

测试经理为新版本增加测试用例时：

1. 打开被测系统并准备账号、环境和测试数据；
2. 启动 Chrome 录制插件；
3. 在页面中完成测试操作；
4. 按需添加可见性、文本、值等断言；
5. 按需添加固定等待或观察等待；
6. 在本地回放并定位失败步骤；
7. 查看、复制或下载生成的 Python Playwright 脚本；
8. 后续由使用者自行保存、维护和执行脚本。

## 4. 已确认需求

### 4.1 输出

- 生成结果必须对用户可见；
- 支持复制和下载；
- 首选输出为标准 Python Playwright pytest 脚本；
- 生成脚本不应依赖 T-Agent 平台才能运行；
- 用户拥有并可直接修改生成结果。

### 4.2 录制与验证

- 支持录制常见点击、输入、键盘、选择等浏览器操作；
- 支持由用户主动添加断言；
- 支持本地回放；
- 回放失败时能够指出失败步骤；
- 支持调节回放节奏，减少页面尚未加载完成导致的失败；
- 同时支持固定等待和观察等待；
- 部分业务场景可能需要等待 3 分钟或更长。

固定等待表示无论页面状态如何，都完整等待指定时间；观察等待表示在最长时限内等待某个元素、文本、URL 或业务状态出现，满足条件后立即继续。

### 4.3 浏览器与使用环境

- 第一阶段只支持 Chrome；
- 公司 Chrome 可能只允许安装内源扩展；
- 每位用户在自己的 Chrome 中录制和回放；
- 不需要服务端浏览器池，也不存在平台侧浏览器并发问题；
- 第一阶段不要求限制可录制的业务域名。

### 4.4 已接受的限制

- 页面 XPath、locator 或元素结构变化后，可以先由人员维护脚本；
- 敏感输入可能以明文进入录制结果，首版可接受，但必须作为后续安全事项跟踪；
- 新标签页或弹窗场景首版不要求；
- 文件上传场景首版不要求；
- 不需要脚本审批；
- 不需要脚本版本历史。

## 5. 产品边界

T-Agent Recorder 与当前项目中的 AI 驱动测试执行是两个独立能力：

- 不复用现有 AI TestSpec、ReAct、MCP 或断言执行链路；
- 不要求把录制结果导入现有 AI 执行模块；
- 不由平台执行生成的脚本；
- 第一阶段不由平台存储、管理或维护生成脚本；
- 不建设多人共享的录制会话；
- 不建设服务端录制与回放基础设施。

## 6. 成功标准

第一阶段成功不以“覆盖所有网页交互”为标准，而以以下结果为准：

- 非编码用户能够独立完成一次录制；
- 用户能够添加关键断言和必要等待；
- 用户能够完成本地回放并理解失败位置；
- 用户能够获得完整、可读、可修改的 Python pytest 文件；
- 工具能够以内部 Chrome 插件形式分发给多个用户；
- 单人可以持续开发和维护该产品。

## 7. 暂不解决的问题

- 生成脚本的集中存储和版本管理；
- 平台调度执行、定时执行或 CI 集成；
- 测试账号、测试数据和环境编排；
- 智能修复全部失效 locator；
- 跨浏览器支持；
- 移动端和桌面客户端录制；
- 团队级脚本协作、评审和权限体系。
