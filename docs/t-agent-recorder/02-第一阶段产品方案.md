# T-Agent Recorder：第一阶段产品方案

> 状态：方案已确认，待技术原型验证  
> 归档日期：2026-07-06  
> 产品形态：独立内部 Chrome 插件

## 1. 决策

第一阶段直接 fork 开源项目 Playwright CRX，建设独立产品 **T-Agent Recorder**。

第一阶段不与 T-Agent 平台集成。平台不作为录制入口，不传递用例信息，也不保存生成脚本。插件在用户本地独立完成录制、断言、回放、代码生成和下载。

这样可以最大限度复用已经验证过的 Playwright Recorder 能力，避免首版同时承担平台通信、鉴权、脚本资产管理和浏览器并发等非核心工作。

## 2. 产品定位

> T-Agent Recorder 是基于 Playwright CRX 的内部 Chrome 录制插件，用于帮助测试人员生成可维护的 Python Playwright 自动化脚本。

它是脚本生成工具，不是自动化执行平台，也不是现有 AI 驱动执行方案的一部分。

## 3. 用户工作流

1. 用户打开被测系统；
2. 完成登录、环境选择和测试数据准备；
3. 点击 T-Agent Recorder 扩展图标；
4. 插件绑定当前标签页并打开原生侧边栏；
5. 用户操作页面，代码在侧边栏中实时生成；
6. 用户通过侧边栏工具栏添加断言；
7. 用户按需添加固定等待或观察等待；
8. 用户停止录制并进行本地回放；
9. 回放日志逐步显示成功、失败和失败位置；
10. 用户查看、复制或下载 Python pytest 脚本。

这里的核心心智是“边操作、边生成代码”，而不是“先录制一份数据，再由平台后台生成脚本”。

## 4. 首版功能范围

### 4.1 保留 Playwright CRX 原生能力

- 绑定当前 Chrome 标签页；
- 录制导航、点击、输入、键盘和原生下拉选择等操作；
- locator 自动生成；
- 元素检查；
- 添加文本、值、可见性、选中状态等断言；
- 实时代码预览；
- 本地回放；
- 回放调用日志和失败步骤；
- 代码复制或文件下载。

### 4.2 T-Agent Recorder 必要改造

- 内部产品名称、图标和说明；
- 公司内部扩展打包与发布配置；
- 默认或仅保留 Python pytest 输出；
- 中文化关键界面和错误提示；
- 增加固定等待；
- 增加观察等待；
- 支持 3 分钟或更长的等待超时；
- 增加回放节奏配置；
- 优化自定义下拉框等常见组件的录制稳定性；
- 按统一规则生成下载文件名；
- 补充许可证、第三方声明和依赖安全检查。

### 4.3 等待能力

固定等待示例：

```python
page.wait_for_timeout(180_000)
```

观察等待优先采用 Playwright 的状态等待或断言超时，例如：

```python
expect(page.get_by_text("处理完成")).to_be_visible(timeout=180_000)
```

观察等待应作为默认推荐，因为条件提前满足后可以立即继续；固定等待用于确实需要完整等待指定时间的场景。

不能把用户录制期间的全部停顿自动转成等待步骤，否则会产生大量缓慢且脆弱的脚本。后续可以对明显长停顿给出添加等待的提示，但应由用户确认等待语义。

## 5. 明确不做

第一阶段不包含：

- 平台入口和平台与扩展通信；
- 平台用例信息展示；
- 用户鉴权；
- 脚本上传和集中存储；
- 脚本版本、审批和团队协作；
- 服务端执行；
- 与现有 AI 执行能力关联；
- 新标签页或弹窗专项支持；
- 文件上传专项支持；
- Edge 或其他浏览器支持；
- 完整结构化步骤编辑器；
- 拖拽排序和低代码流程设计器。

## 6. Fork 策略

- 建立独立内部仓库，例如 `t-agent-recorder`；
- 保留上游 Apache-2.0 许可证及版权声明；
- 锁定 Playwright CRX 和内置 Playwright 版本；
- 将自有修改拆成清晰、独立的提交；
- 尽量保留上游目录结构和测试；
- 不追求实时同步上游，只在安全、兼容或必要功能推动时升级；
- 每次升级先验证录制、断言、回放和 Python 生成主链路。

选择 fork 而不是外围包装的主要原因，是等待能力会同时影响 Action 数据模型、录制器界面、回放 Player 和语言生成器。外围文本拼接无法保证本地回放与最终 Python 语义一致。

## 7. 工程与发布

插件采用 Chrome Manifest V3，由公司内部扩展渠道分发。需要尽早确认：

- 内部扩展商店或 Chrome Enterprise Policy 的发布流程；
- 扩展签名和固定扩展 ID；
- 自动更新机制；
- `debugger` 权限的安全审批；
- 支持的公司 Chrome 最低版本；
- 第三方依赖漏洞治理要求。

录制与回放完全在用户本地进行。多个用户同时使用时，各自使用自己的浏览器资源，因此不产生平台并发或会话隔离问题。

## 8. 技术验证优先级

### P0：决定方案是否成立

1. fork 后能够稳定构建和安装内部扩展；
2. Python pytest 输出完整、可运行；
3. 固定等待进入录制、回放和 Python 生成全链路；
4. 观察等待进入同一全链路，并支持至少 3 分钟超时；
5. 回放失败能够定位到对应等待或操作步骤；
6. 自定义 `<div>/<span>` 下拉框能够通过稳定 locator 录制和回放。

### P1：首版可用性

1. 回放节奏可配置；
2. 中文提示；
3. 下载命名；
4. 崩溃或误关闭后的最小恢复能力；
5. 敏感输入风险提示。

### P2：使用反馈后再决定

- 更丰富的步骤编辑；
- 自动识别长停顿并建议等待；
- locator 重新选择和修复；
- 平台用例上下文集成；
- 脚本资产管理。

## 9. 第一阶段验收建议

选择至少 5 个真实业务场景，覆盖：

- 登录后的普通表单；
- 自定义下拉框；
- 文本、值和可见性断言；
- 短时异步加载；
- 3 分钟以上观察等待；
- 必须完整等待的固定等待；
- 一次故意制造的回放失败。

每个场景需要验证：录制结果、回放结果、失败定位和下载后的 Python 文件。最终验收以下载脚本在标准 Python Playwright pytest 环境中可执行为准。
