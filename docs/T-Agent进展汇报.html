① openclaw 试点效果
openclaw：开源「通用 AI 助手 Agent」（消息/语音/多 Agent 路由 + 浏览器集成），本身不是测试工具。 试点方式：用其浏览器能力直接跑 Web 自动化用例。
✓简单流程能跑通——AI 可自主操作浏览器完成点击/输入
✕判定靠 AI 自述「我觉得成功了」——结果不稳、不可复现
✕一次性执行，不产出可维护测试代码，回归每次重烧算力
✕无断言 / 登录态复用 / 权限 / 调度等测试工程能力
✕Node/TS 个人助手架构，内网部署与合规裁剪成本高
试点录屏（待插入）
② 基于 openclaw 二次开发　vs　自研 AI-Agent
维度基于 openclaw 二次开发自研 AI-Agent（T-Agent）
起步速度快——复用现成 Agent慢——执行/断言/codegen 从 0 搭
架构契合个人助手框架(消息/语音/多Agent)，与测试场景错配，二开=削足适履为测试而生，架构对口
结果判定受其「AI 自述」范式约束，硬塞确定性断言层改造深原生规则引擎确定性裁决
代码沉淀无此概念，需大改原生转译 pytest-bdd 可回放
可控 / 合规依赖上游开源 + Node/TS 栈，内网合规裁剪难完全自主 · Python 栈 · 内网 stdio
维护 / 演进受上游路线牵制，定制易与上游冲突自主演进，贴合内网业务
长期成本二开包袱 + 持续适配上游一次自建、可控
openclaw 定位错配（个人助手 ≠ 测试平台）：二次开发是在错的地基上盖楼，改造成本高且受制于人。 自研虽起步慢，但架构对口、判定与资产沉淀可原生实现、内网可控 → 选择自研。
