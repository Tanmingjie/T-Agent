## 0. 开场:一句话价值主张

【口播】
> "我们在做一个内网测试自动化平台:**测试工程师用业务用例直接驱动 AI 跑测试,
> 跑一次,就自动沉淀出一份可长期维护的自动化测试代码。**
> 目标是把'写自动化脚本'这件高门槛、难维护的事,变成'写业务用例'就够了。"

---

## 1. 背景与痛点

【讲什么】先让领导认同"这个问题值得解"。

【口播】
> "现在的测试执行有两条路,都不理想:
> - **纯手工**:回归测试反复点,人力消耗大、效率低;
> - **写自动化脚本**:门槛高,得懂 Playwright/代码;而且**界面一改,脚本就挂**,维护成本甚至超过收益,很多团队脚本写了一阵就废弃。
>
> 痛点本质是:**自动化的'编写'和'维护'两道坎,把测试工程师挡在门外。**
> 我们要做的,就是用 AI 把这两道坎抹平。"

---

## 2. 我们做了什么(工作流 · 用户视角)

【讲什么】不讲技术,讲"用起来什么样"。一张极简流程图。

【口播】
> "从使用者角度,整个流程就四步:
> 1. **写业务用例**——还是平时那张 Excel(用例名、预置条件、步骤、预期结果);
> 2. **AI 自动执行**——AI 自己打开浏览器,按用例一步步操作;
> 3. **自动判定结果**——系统确定性地判断每条预期对不对,给出可信的通过/失败;
> 4. **自动产出代码**——把这次执行变成一份可维护的标准测试代码,留给团队长期用。
>
> 一句话:**输入业务用例,输出'测试结果 + 测试代码'两份资产。**"

极简流程:
```
业务用例(Excel) → AI 驱动浏览器执行 → 规则引擎判定 → 产出测试结果 + 可维护代码
```

## 3. 方案设计(技术视角)

【讲什么】核心思路 + 五层架构(概览)+ 四段执行流 + 1~2 个技术亮点。
16 步明细、技术实现做**备份页**,有人问再展开。

### 3.1 核心思路

【口播】
> "整个方案有一个贯穿始终的设计原则:**AI 负责'怎么操作',规则引擎负责'对不对',两者分工。**
> 不让 AI 既当运动员又当裁判——AI 去开浏览器、点按钮;但'测试到底通过没有',由确定性的规则引擎来裁决,**结果可复现、可定位**。"

### 3.2 五层架构(一句话记忆:收用例 → 变规格 → 懂页面 → 跑起来 → 留资产)

| 层 | 干什么 | 关键技术 |
|---|---|---|
| ① 输入层 | 把各种来源的用例统一收进来 | Excel 解析 + 预置条件 LLM 分类 |
| ② 执行规格层 | 把人话用例翻译成机器规格 + 结构化断言 | LLM 一次性翻译 → TestSpec |
| ③ 页面智能层 | 看懂页面、模糊词对上真实元素 | 词汇表 + 页面扫描 |
| ④ 执行内核 | AI 真正驱动浏览器跑完用例 | ReAct + 自愈 + 规则引擎断言 |
| ⑤ 输出层 | 把执行变成可留存的资产 | BDD 代码生成 + 持久化 |
| ★ 控制台 | 给人看、给人控 | React + FastAPI + 实时推送 |

### 3.3 执行流(四段,别上 16 步)

【口播】
> "一条用例跑起来,内部分四段:
> **准备**(把用例翻译成规格)→ **启动**(开独立环境、自动登录)→
> **执行循环**(AI 一步步操作、每步验证、出错自愈)→ **收尾产出**(出结果、生成代码、入库)。"

### 3.4 两个技术亮点(只讲价值,不讲实现)

【口播】
> "这里有两点是我们认为最有壁垒、最难做对的:
>
> **第一,判断前移——让结果可信。**
> 业界很多 AI 测试是让大模型'看一眼页面、自己说通过没通过',这非常不可靠:
> 页面上随便哪有'成功'两个字就可能误判,而且两次判断结果都可能不一样。
> 我们的做法是:**把'怎么算通过'在翻译用例时就定成一条条明确规则**,执行时由规则引擎确定性比对。
> 结果**可复现、可定位**,而不是 AI 的一句'我觉得过了'。
>
> **第二,自动自愈——降低维护成本。**
> 传统脚本最大的死穴是界面一改就挂。我们的 AI 在找不到元素时会**自动换策略重新定位**
> (先按语义、再按文本、最后看截图),界面小改动不用人工去修脚本。
> 这正是冲着'自动化维护成本高'这个核心痛点去的。"

## 5. 为什么自研,而不是基于现成方案二次开发(结论页)

【讲什么】回答领导一定会想的"为什么投人力自研"。1 页给结论,细节备份。

【口播】
> "有人会问为什么不直接拿开源 Agent 二次开发。核心是四点:
> 1. **数据不出网**——本地大模型 + 浏览器层走 stdio 绕过公司代理,全程内网,符合安全要求;
> 2. **结果可信**——我们的'规则引擎裁决'是自己设计的核心,开源通用 Agent 大多靠 AI 眼判,达不到测试要的可复现;
> 3. **深度可控**——断言、自愈、词汇表这些测试专有能力,二开受限于上游、改不动核心;
> 4. **资产沉淀**——产出团队能维护的标准 BDD 代码,是我们专门设计的输出层,不是通用 Agent 的目标。
>
> 一句话:**自研的是'测试专有的可信内核',这部分别人替不了。**"
