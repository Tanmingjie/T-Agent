## 1. 产品概述

### 1.1 产品定位

面向测试工程师的**内网 Web 自动化测试平台**。输入业务测试用例（Excel;测试管理平台 API 为阶段五预留），AI Agent 自动驱动浏览器执行，产出可长期维护的 **pytest-bdd Playwright 测试代码**。本地 LLM，私有化部署，数据不出内网。

### 1.2 核心价值

- **降低自动化门槛**：测试工程师用业务用例直接驱动 AI 执行，无需手写 Playwright 代码
- **测试资产沉淀**：AI 执行录制自动转换为高质量 BDD 测试代码，供长期维护
- **内网友好**：playwright-mcp stdio 模式绕过公司代理，本地 LLM 数据不出网
- **自愈能力**：界面变化后 Agent 自动重新定位元素，减少人工维护选择器的成本
- **可信断言**：预期结果转为结构化断言，由规则引擎验证，PASS/FAIL 可复现可定位

### 1.3 边界约束

| 类别 | 内容 |
|---|---|
| 不做 | 多人协作/账号体系、CI/CD 集成、TypeScript 代码生成、公网 LLM、跨浏览器 |
| 暂不做（预留接口） | 测试管理平台 API 对接、Page Object 模式、报告导出、失败断点续跑 |
| 已实现（v2.0 曾列暂不做） | **并发执行**——Orchestrator 支持 `parallelism` 可配并发，默认 1=串行 |
| 核心约束 | 私有化部署、本地 LLM、playwright-mcp stdio、断言由规则引擎裁决（非 LLM 眼判） |

## 2. 整体架构

### 2.1 五层架构

```
① 输入层      Excel（API 为阶段五预留）→ TestCase + 预置条件分类
② TestSpec    结构化执行规格 + 断言翻译
③ Page Intelligence  词汇表 / 操作图谱 / 预解析
④ Agent Harness      ReAct + 自愈 + 断言验证
⑤ 输出层      BDD 代码 + 执行记录
              ┕━ 之上：工程化界面 Platform UI
```

### 2.2 完整执行流（16 步）

| 步骤 | 动作 | 涉及的 Harness 机制 |
|---|---|---|
| 1 | Excel 导入 | openpyxl → TestCase 结构体 |
| 2 | 预置条件分类 | LLM 三分类：状态声明→Hook / 操作→Given / 模糊→确认 |
| 3 | 用例预解析 + 断言翻译 | 结合词汇表生成 TestSpec，预期→结构化断言 |
| 4 | 启动 TestCase Subagent | **Subagent 隔离**：每用例独立 context |
| 5 | before_case Hooks | **Hooks**：init_env_aw → LoginHook（Cookie 复用） |
| 6 | Prompt 组装 + Skill 注入 | **Skill**：Base + DomainSkill + Task(TestSpec) + ToolSkill |
| 7 | Reason | **ReAct**：StepPlan + A11y Tree(截断) + PageSkill → 决策 |
| 8 | Permission 检查 | **Permission**：高危词/prod → 暂停确认 |
| 9 | Act | **ReAct**：MCP/Custom Tool。录制 ActionStep |
| 10 | Observe | **ReAct**：A11y + URL + 截图。PageSkill 切换 |
| 11 | 断言验证 | **Assertion**：规则引擎验证（非 LLM 眼判） |
| 12 | Context Compact + 循环检测 | **Context**：步骤归档、A11y 截断、Skill 卸载 |
| 13 | Self-Healing（失败时） | **Healing Subagent**：独立 context，P1→P5 + 截图 + 词汇表 |
| 14 | after_case Hooks | **Hooks**：清理、退出登录、on_failure |
| 15 | CodeGenerator → BDD 代码 | TestSpec + 录制 → feature + step_defs。断言→expect() |
| 16 | 持久化 + Orchestrator 汇总 | 入库、截图存盘、词汇表增量、Suite 报告 |

## 3. 输入层

### 3.1 Excel 用例格式

| 字段 | 说明 | 示例 |
|---|---|---|
| 用例名称 | 唯一标识 | 采购申请单提交验证 |
| 预置条件 | 混合态：状态声明 + 操作步骤 | 已登录系统；环境已装好；设置环境变量 CONF=10 |
| 测试步骤 | 有序步骤（1./①/换行），**数据写死在步骤里** | 1. 进入采购申请页面 2. 标题填"测试采购单" 3. 点击提交 |
| 预期结果 | 翻译为结构化断言 | 申请单状态变为"待审批"，显示提交成功提示 |

> **说明**：测试数据直接写死在步骤文本里，平台不做数据工厂/动态生成。

### 3.2 预置条件三分类

混合态用 LLM 意图分类（**不用关键词匹配**，成功率低）。

| 类型 | 示例 | 处理方式 |
|---|---|---|
| ① 状态声明 | "已登录系统" / "环境已装好" | 映射 Hook，平台自动处理（规则可维护） |
| ② 操作步骤 | "设置环境变量 CONF=10" | 转为 Agent Given 步骤 |
| ③ 模糊/低置信 | "准备好测试环境" | 标黄，用户手动选择 Hook/Given/忽略 |

> **重要**：分类结果首次确认后记忆到用例，下次跳过。可随时修改。

## 4. TestSpec（执行规格）【v2 新增】

### 4.1 定位：软计划，不是硬脚本

TestSpec 是"用例文本"和"Agent 执行"之间的**结构化执行规格**，统一了三件事：预解析输出 + 用户确认对象 + 代码生成输入。

> **关键**：TestSpec 记录"动作 + 目标语义 + 数据 + 断言"，但**不锁定具体 selector**（执行时词汇表/自愈动态解析）。是"执行计划"不是"执行脚本"，保留 AI 弹性。

### 4.2 结构示例

```yaml
case_id: "TC001"
name: "采购申请单提交验证"
base_url: "http://erp.corp.com/purchase"
given:                          # 来自预置条件②类
  - action: set_env
    data: "CONF=10"
steps:
  - action: navigate
    target: "采购申请页"
  - action: fill
    target: "标题输入框"        # 词汇表解析
    data: "测试采购单"          # 步骤里写死
  - action: click
    target: "提交"              # 不锁 selector
assertions:                     # 来自预期结果
  - type: text_equals
    target: "状态字段"
    expected: "待审批"
  - type: element_visible
    target: "提交成功提示"
```

### 4.3 收益

执行更稳定（决策前置）、可审查可修正、代码生成更精确（从 Spec 而非录制噪声）、可复用可版本化。

## 5. 断言机制【v2 核心强化】

### 5.1 核心思路：裁决以偏-FAIL LLM 裁判为主 + 确定性锚点（2026-06-17 据真实评测重订）

> **重订说明**：原方案"把判断移到翻译时、由规则引擎确定性验证为主"在真实用例上**实测失败**——
> 见下「实测结论」。本节据真实公网评测改写;早先版本的"LLM 眼判不可靠"假设未被数据支持。

**实测结论（automationexercise 公网评测）**：
- **结构化规则引擎在真实页面贡献≈0**：翻译几乎不产出结构化断言（必填的自然语言判据挤掉了可选结构化断言）;业务词↔真实元素定位脆弱（跨语言、多相似元素），**false-fail 高到用例不可用**。
- **担心的 false-green 没兑现**：26 条真/假混合预期评测，偏-FAIL 的 LLM 裁判 **false-green=0/15、正确率≈96%**;唯一 1 例假绿来自门控 fail-open + JSON 解析炸（工程缺陷），非"脑补成功"。

**方案**：裁决以 **偏-FAIL 的 LLM 裁判为默认主裁决**（强制引证页面证据、拿不准判 FAIL、标 `ai_judged`），辅以**确定性锚点**（URL / 数据真值 custom_tool / 显式 selector，能用则优先,作高置信证据）。判断在**终态**做,不前置锁死;裁决路径 **fail-closed**。

> **更新（2026-06-24,用户拍板①）**:此处"强制引证页面证据"指**模型自身在 evidence 字段逐字摘录**
> （偏-FAIL 纪律 + reason 可审计),**平台不再对该 evidence 做确定性接地核验/推翻**。Fix 3 收尾曾
> 加过"平台核验 evidence 真在页、不在则 fail-closed 推翻"的接地层,但 `eval_fg/ab_grounding.py`
> A/B 扩样实测(deepseek-v4-flash,n=63,3 站点)**该层净 ≤0**(有益拦截恒 0、仅偶发误伤)→ **整层
> 撤销,裁决权交回模型**。fail-closed 现仅指**解析卫生**(无 verdict/调用失败/未接 LLM → FAIL)。
> **勿据"强制引证"重新加回平台接地核验**;回归基准在 `eval_fg/`。详见 `CLAUDE.md`「撤销证据接地推翻」。

```
预期结果(自然语言)
  ├─ 能查真值/URL/selector → 确定性锚点(高置信)   ┐
  └─ 其余(UI 态)         → 偏-FAIL LLM 裁判(引证+ai_judged) ┘ → 合并裁决(任一FAIL不过) + 绿色抽检
```

### 5.2 五类结构化断言（按可靠性排序）

| 类型 | 可靠性 | 能力 | 示例 |
|---|---|---|---|
| ① DOM 元素 | 最高 | 存在/可见/可点击/数量/状态 | `element_visible(role=status name=待审批)` |
| ② 文本内容 | 高 | **限定具体元素内**匹配 | `text_equals([data-field=status], "待审批")` |
| ③ URL/导航 | 高 | URL 等于/包含/匹配 | `url_contains("/approval/pending")` |
| ④ 数据(Custom Tool) | 业务级 | 数据库查询/API 校验 | `db_query(order_id, status=待审批)` |
| ⑤ LLM 语义 | 兜底 | 无法结构化时最后手段 | `llm_judge("显示成功提示")` |

> **关键（2026-06-17 重订）**：上表"可靠性"是**理论排序**;真实评测里 ①②（DOM/文本）因业务词↔元素定位脆弱而 false-fail 高、实际不可靠。**新策略**：能用 URL/数据真值/显式 selector 的预期 → 作**确定性锚点**(高置信);其余 UI 态预期 → **偏-FAIL LLM 裁判为主**(不再是"最后兜底",而是默认主裁决,实测 false-green≈0)。文本断言仍限定元素内匹配以减少"全页误判",但不作主裁决。

### 5.3 翻译规则

| 预期结果（原文） | 翻译为 Assertion | 说明 |
|---|---|---|
| "状态变为待审批" | text_equals(status元素, "待审批") | 词汇表定位，限定范围匹配 |
| "提交成功，跳转列表页" | url_contains("/list") AND element_visible(成功提示) | 一对多拆解（AND） |
| "列表新增一条记录" | element_count(列表行, increased) 或 db_query | UI 计数或数据库验证 |
| "页面显示错误提示" | element_visible(role=alert) | 语义角色定位 |

### 5.4 断言与自愈的关系

断言失败两种归因：**真失败**（操作执行了但业务结果不对 → FAIL）/ **断言 selector 失效**（界面变了 → 触发自愈重定位断言目标）。Healing Subagent 同时服务于"操作定位"和"断言定位"。

### 5.5 裁决规则【v2.1 明确】

最终 PASS/FAIL **以断言裁决为准，不取 LLM 自报的 TEST_RESULT**（铁律）。聚合规则：

| 断言状态 | 含义 | 对裁决的影响 |
|---|---|---|
| PASS | 确定性比较通过 | 计入通过 |
| FAIL | 比较失败 / 真失败 | **任一 FAIL → 用例不通过** |
| healable | 目标找不到，待自愈复验 | 自愈成功转 PASS/FAIL，失败超限记 FAIL |
| ai_judged | **偏-FAIL LLM 裁判**(默认主裁决,强制引证页面证据) | **计入裁决**;标记可见、绿色抽检使 false-green 可回溯 |
| skipped | `custom_tool` 未注册 / 裁判失败无证据等无法验证 | **不静默放过；全 skipped 不算可信通过**;裁决路径 **fail-closed**(裁判解析失败→FAIL/skipped,不默认绿) |
| (门控判定) | 偏-PASS 步骤门控 `_gate_step_done` | **仅驱动信号,不计入裁决**(它 fail-open,计入会造假绿) |

> **更新(2026-06-22 阶段化 + 2026-06-23 G1,本表部分超期)**:阶段化重设计后裁决改为**逐阶段
> Validator**(`on_phase_end` 在阶段边界跑 `_check_llm_judge`),`_gate_step_done` 步骤门控**已删除**
> (门控行作废);llm_judge 路径的 **`skipped` 已退役**——G1 把"未接 LLM / 调用失败 / 解析不出
> verdict"三态一律收成 **FAIL**(阶段化下 LLM 是主裁决、自动化平台无人工复核,缺失不默认绿)。
> `skipped` 现仅余 `custom_tool` 未注册等**非阶段化路径**。当前裁决底线见 `CLAUDE.md`「撤销证据
> 接地推翻」+「G1」。

> **裁判分工（2026-06-17 重订）**：裁决用 **偏-FAIL 的 LLM 裁判**（`_check_llm_judge`,引证证据、拿不准判 FAIL、fail-closed），**不再是"末档兜底"而是默认主裁决**（实测 false-green=0/15）。**能用 URL/数据真值/selector 的预期优先作确定性锚点**（高置信）。**偏-PASS 的步骤门控只驱动执行、不计入裁决**。来源去重仍按 `agent.collect_assertions`（同一断言别既放用例级又放 step）。

## 6. Page Intelligence 层

### 6.1 设计目标

解决**业务用例文本质量差**：模糊词（"提交"）映射到真实 UI 元素（"保存并提交"）。同时辅助自愈重定位、辅助断言翻译。

### 6.2 三个核心能力

| 能力 | 说明 | 可维护 |
|---|---|---|
| 词汇表 | 业务词→UI元素映射 | 可增删改，手动条目优先级更高 |
| 操作图谱 ⚠️**预留(未实现)** | 按功能区分组的可交互元素 | 可标注业务含义，显示 stale |
| 用例预解析 | LLM + 词汇表生成 TestSpec | 默认手动确认，歧义词高亮 |

> ⚠️ **操作图谱当前为预留**:`PageVocabulary.action_map` 仅是占位字段、无语义定义与执行接入
> (与 §1.3 口径一致)。现已落地的是「词汇表 + 用例预解析」两项;操作图谱待后续阶段设计。

### 6.3 扫描策略 A+C

- **A（手动首次）**：用户触发，选 Session Profile，扫描 base_url + 关键子页面
- **C（执行中增量）**：`agent.run` 结束后复用 ReAct 期间已捕获的 a11y 快照，按 URL 去重、
  独立 context 提炼业务词→元素映射并落库（**手动条目优先**，AI 标 `source=ai`）。
  扫前先 `find_page` 查重，非 stale 就跳过，避免同界面多用例重复提炼。

> **冷启动**：首次执行若词汇表为空，提示先 Page Scan。
> **词条形态**：除 `role+name`（a11y）外，支持 **selector 型词条**（CSS），对计数角标等
> 用 `browser_evaluate` DOM 求值更稳健。解析优先级：显式 selector > 词汇表 selector >
> 词汇表 role+name 精确 > 原始 a11y。
> **维护闭环**：词汇表落 DB，前端 `VocabularyPage` 可增改删（页面/词条/selector），
> 执行期 `VocabularyResolver` 注入 Agent，真正"维护→执行用上"。

### 6.4 缓存设计

Key = `url_pattern + page_title + login_role`。路由匹配（/order/{id}）。更新触发：手动 / stale / 定时。登录态隔离。

> **宽松匹配**：运行时 `page_title` / `login_role` 常未知，`find_page` 对空值做宽松匹配，
> 让手动词条在 role 未知时也能命中。

## 7. Agent Harness 层

### 7.1 Session Profile

账号 + 登录脚本 + Cookie 缓存打包，Suite 绑定，Scanner 与 Hook 共用。

| 字段 | 说明 |
|---|---|
| name | Profile 名称 |
| login_aw | 登录 AW 引用（已有 pytest 脚本） |
| cookie_store | Cookie 持久化路径 |
| valid_until | 有效期，过期自动刷新 |

> **复用**：before_case 检查 valid_until，未过期注入 Cookie 跳过登录。有效期内跨用例共享。

### 7.2 ReAct 执行模式

| 步骤 | 说明 | 录制 |
|---|---|---|
| Reason | 读 StepPlan + A11y Tree(截断) → 决策 | reasoning |
| Act | 执行 MCP/Custom Tool/Hook | tool_name/input/intent |
| Observe | A11y + 截图 + URL，更新 StepPlan | tool_result/screenshot/url |

**StepPlan（TodoWrite）**：TestSpec.steps → 状态机 pending/active/done/failed/skipped，防跳步/遗漏/重复。

**护栏（对本地/弱模型尤为关键）【v2.1 补】**：
- **每轮强制一个工具调用**：Base Prompt 约束"回复必须以恰好一个工具调用结束（`mark_step_done` 也算）"，治弱模型"叙述代替动作"。
- **哑火续推**：模型只回文字不发 tool_call 时，主动抓最新完整快照作普通 user 消息回灌 + 强指令逼出单个工具调用（含具体 ref），超 `max_idle_nudges` 才终止。
- **循环检测 / max_steps**：连续 3 次同调用判定卡死。
- **不采信自报结果**：有未完成步骤时即便模型自报 `TEST_RESULT` 也不终止，改哑火续推（贯彻"裁决以断言为准"到执行层）。
- **观察文本回灌**：观察以 user 消息文本回灌，不依赖严格 `tool_call_id` 配对（本地模型更稳）。

### 7.3 Self-Healing

触发：元素未找到、操作无效、连续重复、超时、**断言 selector 失效**。

多策略定位优先级：P1 语义角色 → P2 文本 → P3 属性 → P4 位置关系 → P5 截图视觉。

流程：抓快照+截图（双通道）→ LLM 生成 3 候选 → P1→P5 尝试 → 成功回写 ActionStep+词汇表 / 失败超限 FAIL。

> **核心原则**：ActionStep 存"操作意图"而非只存选择器。词汇表第一优先查询。
> **防臆造【v2.1 补】**：LLM 生成的候选**必须落在当前快照里**，杜绝凭空发明选择器；
> 操作侧与断言侧自愈均接入词汇表（先 `vocab_resolver.resolve` → 真实页面名作 P1 候选）。

### 7.4 Subagent（上下文隔离）

按**用例级**隔离（非步骤级——用例内步骤有状态依赖）。

| 类型 | 作用 | 优先级 |
|---|---|---|
| 用例 Subagent | 每用例独立 context，批量执行互不污染 | **P0 必做** |
| 自愈 Subagent | 失败尝试在独立 context，主 context 只拿摘要 | **P1 强烈建议** |
| Scanner Subagent | 扫描独立 context，A11y 数据不污染执行 | **P1 自然实现** |

> **边界**：用例内步骤之间不拆 Subagent，状态必须同 context 传递。

### 7.5 Context Compact

- L1 步骤归档：done 步骤移出 context，归档一行摘要
- L2 A11y 截断：注入前相关度过滤（借鉴 browser-use）
- L3 Skill 卸载：离开页面卸载 PageSkill
- 自愈特殊：自愈中不压缩，成功后立即归档

### 7.6 System Prompt 分层 + Skill

| 层级 | 内容 | 来源 |
|---|---|---|
| Base | 角色/ReAct/Behavioral Nudges/TEST_RESULT/安全边界 | 平台固定 |
| Context | 业务知识，{{变量}}插值 | Case>Suite>全局 |
| Task | TestSpec/StepPlan/Hook 状态 | 每次注入 |
| Tools | MCP+自定义工具（相关度截断） | 动态生成 |

三种 Skill：DomainSkill（Suite级）、PageSkill（URL动态加载）、ToolSkill（相关度过滤）。

### 7.7 Hooks

| 时机 | 用途 | 失败处理 |
|---|---|---|
| before_suite | 全局初始化 | 终止整批 |
| before_case ★ | init_env_aw + LoginHook | 用例 FAIL |
| after_case ★ | 清理、退出登录 | 记录，不影响结果 |
| on_heal | 自愈通知 | 仅通知 |
| on_failure | 截图告警 | 仅通知 |
| after_suite | 汇总报告 | 记录 |

### 7.8 Custom Tools

LLM **按需调用**（区别于 Hook 强制执行）。接入：Python @tool 装饰器 / YAML 配置。场景：数据库断言、API 校验、通知。

> **Hook vs Tool**：Hook = 必须执行的前置/后置（初始化、登录）；Tool = LLM 按需调用的能力（数据断言、API 校验）。

### 7.9 Permission

高危操作白名单（删除/提交/支付）+ 环境保护锁（prod URL）。触发位置：Reason 后 Act 前。asyncio.Event 暂停 + UI 弹窗。可配信任模式。

## 8. 输出层

### 8.1 CodeGenerator 抽象层

统一接口 `generate(record) → str`。BDDGenerator（默认）/ PlainGenerator / PageObjectGenerator（预留）。

### 8.2 BDD 代码生成规则

| BDD | 来源 | 说明 |
|---|---|---|
| Given | 预置②类 + Hook 状态 | TestSpec.given |
| When | TestSpec.steps（业务粒度） | 按用例步骤合并 |
| Then | TestSpec.assertions | **断言映射 expect()** |

选择器优先 get_by_role/get_by_label。可读性 > 简洁性，中文注释。验证：ast.parse + black + 自动 pytest 回放（**已 live 验证**：saucedemo 登录/加购/完整结算生成代码均经 pytest-playwright 真回放通过）。

> **框架无关定位器解析层【v2.1 补，`codegen/locators.py`】**：稳健度 `ROLE>TEST_ID>LABEL>PLACEHOLDER>TEXT>CSS`。解析放在 generator **之外**，各 CodeGenerator 只渲染自身语法（BDD 只是一种实现）。
> **Ground truth 优先**：优先用 ReAct 执行时**实际跑通的 Playwright 定位器**（从 tool_result 抓取）重建 Locator，胜过快照重建的可能歧义 role+name（如 6 个"Add to cart"）。
> **遗留边界**：断言目标（如购物车角标）走 probe，不产出可复用选择器，无 vocab 时仍文本兜底。
> **隐式导航**：用例把"打开页面"写在预置条件（被归 state_hook 不进 steps）时，codegen 前置注入 `page.goto` 隐式导航步，保证回放能开页面。

### 8.3 执行记录存储

执行总览（含 token 用量、自愈次数）、步骤时间线（reasoning 可展开、断言结果）、截图序列、生成代码（在线编辑/下载，步骤代码双向关联）。

## 13. 工程化界面设计【v2.1 新增】

工程化界面是五层架构之上的 **Platform UI**，目标：让"维护用例/词汇表 → 执行 → 看清全过程 → 拿到产物"形成闭环，且**执行态可观测**（实时看到 Agent 在干什么）。

### 13.1 前后端边界

- **纯 API 后端**（FastAPI `:8000`）：所有能力经 HTTP 暴露，**不挂前端静态构建**——根除"改了前端但后端服务旧 `dist`"的反复混乱。
- **前端**（React + Vite + Tailwind，`:5173`）：只调 API。即使单机也视为独立两端（呼应实现原则#1）。
- **实时通信**：SSE 单向推送执行进度，自动重连。

### 13.2 信息架构

| 层级 | 页面 | 职责 |
|---|---|---|
| 全局（浅色侧栏） | 测试套件 / 词汇表 | `RootLayout` |
| 套件内（面包屑+导航） | 用例 / 执行历史 / 设置 | `SuiteLayout` |
| 用例详情 | **双栏抽屉** `CaseDrawerBody` | 左=信息/结果卡/步骤卡；右=最终态截图 / 代码 / 断言结果 / 单步截图 |
| run 详情 | `SuiteRunDetailPage` | 点用例**复用同一抽屉**（绑定该次 runId），不三页跳转 |

> 用例页支持**原地执行**（`useSuiteRun` 封装 SSE），不再跳独立控制台页。

### 13.3 执行态可观测（本平台关键差异点）

弱模型执行一条用例耗时长，"执行中抽屉空白"是必须解决的体验问题。设计：

- **即时步骤推送**：`ReActLoop.on_step` 每步落定**即时**发 `step_change`（非事后补发）。
- **生命周期阶段**：`agent.run` 发 `phase` 事件（spec 翻译 / executing / asserting / codegen）；前端 `RunningView` 渲染阶段清单（末项转圈、其余打勾）+ 实时步骤。
- **翻译产物实时可见**：翻译完成即发 `spec_ready`，执行中点"用例信息"就能看 TestSpec。
- **状态区分色**：running 用蓝、通过才绿（不再全程一片绿）。
- **步骤截图按真实落图状态**：`step_change` 带真实 `screenshot` 文件名，失败/重试/快照步本不落图，不再误取不存在的图报 404。
- **查看 prompt**：每步可展开当轮请求（落库 + 执行中实时都能看）。

> SSE 事件全集：`suite_start / case_start / phase / spec_ready / step_change / case_result / suite_done`（+ 权限 `permission_required`）。

### 13.4 执行隔离（工程化必修）

> **背景**：uvicorn 单进程/单 loop，长任务（用例执行）跑在 API 共用 loop 上时，任何不让出的同步代码都会让**所有 HTTP 请求 pending**（表象像 DB 锁，实为 loop 被占）。

设计：执行**整体搬出 API loop**——每个 run 一守护线程 + 独立 event loop + 独立 Store（SQLAlchemy async engine 绑定创建它的 loop，跨 loop 各用各的）；SSE 经 `call_soon_threadsafe` 桥接回 API loop；权限审批走 `threading.Event` 跨线程。dev 启动器 `scripts/serve.py` 的 `--reload` 只监视源码目录，避免 codegen 写 `storage/generated/*.py` 触发后端重启打断 run。

### 13.5 产物可视化

抽屉的断言/代码/截图均来自 `ExecutionRecord`（per-run 数据存进记录，不按 case_id 落平文件以防被后续 run 覆盖串味）：

- **断言结果**：聚合用例级 + 步骤级 expect，与 `collect_assertions` 一致；标 PASS/FAIL/skipped。
- **生成代码**：浅色主题 + 行号 + 限高滚动 + 复制；`/code` 优先返回 per-run `generated_code`。
- **截图**：浏览器动作后落 `step_NNN.png`（真实 run_id 目录）；无图回退"无截图"。

> **注意**：结果接口 `results.py::_build_history` 与 `recorder.to_history` 是**两套序列化**，
> 改 history 字段需两处同步（否则"执行中能看、完成后丢"）。

### 13.6 设计语言

TestSprite 风格：森林绿/沙绿强调色（`brand` 令牌）、浅色分组侧栏、表格列表、双栏抽屉、`canvas` 灰底白卡。Design Tokens 集中在 `tailwind.config.js`（`brand` / `surface` / `shadow-card` / `shadow-elevated`）。
