# AI Web 自动化测试 · 业界调研(2025–2026)

> 一份关于「AI / LLM 驱动的 Web 端到端自动化测试」的独立行业调研。
> 范围:自然语言测试平台、AI 自愈测试、自主浏览器 Agent、Computer-Use Agent,以及学术 benchmark 的真实表现。
> 数据来源见文末 Sources;基准数字以公开论文 / 评测榜 / 第三方评价站(G2、Capterra)为准,厂商自报口径单独标注。

---

## 摘要(TL;DR)

- **纯 AI 自主测试的真实天花板:复杂真实流程约 50–60%(最强),平均约 30%。** 实验室 benchmark(WebVoyager)上 ~90% 的成绩,换成真实网站 + 严格人工评分后大幅下滑。
- **商用做得最好的产品,靠的不是"更强的 AI",而是"AI + 人在环 + 托管基建"。** 满意度最高的 QA Wolf(G2 4.8)本质是人机协同的托管服务,而非纯 AI。
- **行业共识:AI 能生成测试代码,但生成不了"测试判断"(断言 / 业务逻辑)。** 这是全行业最难、最值钱的一环。
- **连"测准确率"本身都不可靠:** 同一批 Agent,用 LLM-as-judge 自动评分能显示 80%+,用严格人工评分只有 30–60%——评分方法能把数字摆动 50 个百分点。
- **信任在下降:** 开发者对 AI 输出的信任从 2024 的 69% 跌到 2025 的 54%;67% 的测试人员只接受"带强制人工复核"的 AI 测试。

## 一、赛道地图:四个层次

「AI Web 自动化测试」不是一个东西,而是四个相互叠加的层次:

### 1. 商用自然语言 / 自愈测试平台(面向 QA 团队)
| 产品 | 路线 | 定位 |
|---|---|---|
| **testRigor** | 纯自然语言写测试,AI 适应 UI 变化 | NL 入门最知名 |
| **mabl** | GenAI 自愈、从用户故事 / Jira 生成测试 | 老牌企业级,主打降维护 |
| **Functionize** | agentic 平台,NL 生成 + 视觉识别 + 运行时自愈 | 闭源云 |
| **Momentic**(YC) | AI-native 低代码,NL 断言 | 专攻 AI 应用的非确定性 |
| **Checksum** | 观察生产环境真实用户行为 → 生成测试 | production-traffic 路线 |
| **Testim(Tricentis)/ Katalon / Applitools** | 传统厂商的 AI 自愈 / 视觉 AI | 存量市场 AI 化 |

### 2. 托管式 AI QA 服务(人 + AI)
| 产品 | 模式 |
|---|---|
| **QA Wolf** | 人类 QA 工程师 + AI,托管建/维护/跑 E2E,承诺高覆盖近零 flake |
| **Octomind** | 给 URL 自动发现并生成 Playwright E2E,托管基建 |
| **Rainforest QA / Applause** | 众包 / 托管测试 |

### 3. 开源浏览器 Agent 引擎(执行层框架)
| 项目 | 路线 | WebVoyager 自报 |
|---|---|---|
| **browser-use**(50k+ stars) | DOM,把任意 LLM 变浏览器 Agent | **89.1%** |
| **Skyvern** | 视觉(CV + LLM),擅长填表 | **85.85%** |
| **Stagehand**(Browserbase) | TypeScript,生产级 Agent SDK | — |
| **LaVague** | 开源 web agent | — |

### 4. 通用 Computer-Use Agent(最上游,不专做测试)
- **OpenAI Operator**、**Anthropic Claude Computer Use**、**Google Project Mariner** —— 看屏幕 / 操作鼠标键盘的通用代理。

## 二、能做到什么程度?—— Benchmark 真实表现

### 2.1 实验室成绩 vs 真实成绩

| 评测集 | 性质 | 最强成绩 |
|---|---|---|
| **WebVoyager** | 少数网站、任务有捷径、自动评分宽松 | browser-use **89.1%** / Skyvern **85.85%** |
| **WebArena** | 模拟站点 | OpenAI Operator **58.1%** |
| **OSWorld**(OS 级任务) | 真实桌面任务 | Operator **38.1%** |
| **Online-Mind2Web** | **300 任务 / 136 个真实网站 / 严格评分** | 平均 **~30%**,最强(Operator / Claude Computer Use)**~50–60%** |

### 2.2 关键发现:《An Illusion of Progress?》(进步的幻觉?)

俄亥俄州立 NLP 组的 **Online-Mind2Web** 论文(标题即《An Illusion of Progress? Assessing the Current State of Web Agents》)指出:

> WebVoyager 上 ~90% 的成绩,搬到真实动态网站、严格评分后**崩到 30% 量级**;而且**很多"新" Agent 还不如 2024 年初的简单基线 SeeAct**——堆模型 / 堆框架并没有换来真实可靠性。

基准细节:300 个任务、136 个真实网站,覆盖服饰 / 餐饮 / 住房 / 出行等真实活动;同时采用人工评估与 LLM-as-judge。

### 2.3 元发现:连"测准确率"本身都不可靠

同一批 Agent,**评分方法不同,结果摆动巨大**:
- 用 LLM-as-judge(如 WebJudge / GPT-4o、o4-mini 为评委)自动评分:可显示 **80%+**;
- 用严格人工评估:真实复杂任务只有 **30–60%**。

> 含义:当"裁判"是 LLM 时,成绩单是可以被注水的(false green)。这使得"AI 测试的准确率"这个数字本身需要警惕评分口径。

### 2.4 小结

> **纯 AI 跑真实复杂业务流程,最强水平约 50–60%,平均约 30%。没有谁"解决"了这个问题。** 这正是为什么没有团队敢拿纯 Agent 做无人值守的回归门禁。

## 三、真实情况:扒真实评价

### 3.1 巨大的"炒作 vs 落地"鸿沟
- **75% 的组织把 AI 测试列为 2025 战略核心,但只有 16% 真正落地**——59 个百分点的鸿沟。
- Reddit r/QualityAssurance 上"有人真在 QA 里用上 gen AI 吗?"的讨论,多数回答是"在试,但很少真上生产"。

### 3.2 信任在崩,且越资深越不信
- 开发者对 AI 输出的信任:**2024 年 69% → 2025 年 54%**;只有 **33% 真正信任**;**仅 3% 自称"高度信任"**。
- **"最资深的工程师最怀疑"**(10 年以上经验者信任最低),早期职业开发者最热情——*"怀疑是试出来的"*。
- **67% 的测试人员只接受带强制人工复核的 AI 测试。**

### 3.3 哪些真能用 / 哪些一直翻车(从业者共识)

| ✅ AI 确有价值 | ❌ AI 一直做不好 |
|---|---|
| 生成脚手架 / 样板代码 | **生成有意义的业务逻辑断言** |
| 测试命名、测试数据(含边界) | 端到端测试设计 |
| **自愈 selector**(应对 UI 变动) | 复杂集成测试 |
| 根因分析辅助 | 安全 / 性能测试场景 |

> 一句被反复引用的结论:**"AI can generate test code. It cannot generate test judgment."**(AI 能生成测试代码,但生成不了测试判断——因为断言需要理解业务逻辑与上下文。)
> 另一组数据:人工复核 AI 输出**平均每周约 4.3 小时 / 人**——"自动化"省下的时间会被复核吃回去一部分。

### 3.4 逐产品口碑(G2 / Capterra)

- **testRigor**(G2 4.7):自然语言上手好、客服响应快、文档友好;但**复杂用例时灵活性受限、定制受约束**——NL 的甜蜜区在简单到中等流程。
- **mabl**:自愈能力被普遍称赞("auto-healing 对 UI 变化很有用");但**"比同类贵"**是高频吐槽;亦有用户反馈**录制不准、首次回放失败、坏掉的测试没有便捷修复路径**。
- **QA Wolf**(G2 4.8):好评集中在**"像我们 QA 团队的延伸"、7×24 客服、显著提升发布信心、覆盖率从很低拉到很高**;吐槽是**贵、随测试量增长成本上升、跑得慢、外部团队拿不到一线痛点**;**平均 2 个月落地、8 个月回本**。

### 3.5 一个值得注意的"潜台词"

> **满意度最高的产品(QA Wolf)是靠"人在环"做到的,不是靠纯 AI 全自动。** 这本身就是行业对"纯 AI 还不够可靠"的诚实承认:在可靠性这条线上,目前最优解仍然是 **AI 提效 + 人类把关 + 托管基建**。

## 四、深度分析:两个标杆

"最好"分两种定义,各拆一个。

### 4.1 标杆 A：QA Wolf —— 商用最佳(可靠交付)

**是什么:** 不是一个工具,而是**"测试即服务"(testing-as-a-service)**。客户提供应用,QA Wolf 的**人类 QA 工程师 + AI** 帮忙建立、维护、运行 E2E 测试,承诺高覆盖率、近零 flake,失败时有人协助定位。

**为什么它"最好"(模式拆解):**
1. **人在环兜底** —— AI 生成 / 维护测试,人类工程师审一道,用人力补上"AI 不可靠"这个行业病。这是其高满意度的根本。
2. **托管基建** —— 并行运行大规模套件而不引入 flake(基建层吸收不稳定性),这是纯软件工具难以提供的。
3. **卖的是"可信结果"** —— 核心卖点不是"AI 多聪明",而是**"绿就是真绿"**,因此客户敢拿它做发布门禁。

**代价 / 局限:**
- **贵 + SaaS 云 + 数据出门** —— 代码 / 测试数据需交付平台,**强监管 / 内网 / 政企场景受限**。
- **黑盒** —— 客户不拥有引擎,也"拿不到一线痛点"。
- **落地慢** —— 平均 2 个月起步、8 个月回本。

**结论:** QA Wolf 证明了**"AI + 人在环 + 可信结果"是当前最务实可靠的路线**,但它把"人在环"做成了**卖人力的托管服务**。

### 4.2 标杆 B：OpenAI Operator / Anthropic Claude Computer Use —— 纯 AI 自主最强

**水平:** 真实网站严格评分约 **50–61%**,是所有自主 Agent 的天花板;**但仍有约 40% 失败率**。

**强在哪:**
- 通用 computer-use(看屏幕 / 操作鼠标键盘),不限网站、不依赖预设脚本;
- 大厂顶配多模态模型 + 大规模训练。

**为何仍不够:**
- **40% 失败率**在测试场景是灾难——测试要的是确定性,而非"大概能点对";
- 通用 Agent **不产出可信裁决**:它能"完成任务",但判不了"这次是 PASS 还是 FAIL"(回到"生成不了测试判断")。

**结论:** 纯 Agent 是能力上限的代表,但**通用"能完成任务" ≠ 测试需要的"能确定性裁决"**,二者之间的鸿沟是当前自主测试无法无人值守的核心原因。

---

## 五、横向结论

1. **能力天花板低且无人越过:** 真实复杂流程,纯 AI 最强 50–60%、平均 30%;商用最佳靠人兜底。
2. **可靠性的当前最优解是"AI 提效 + 人类把关 + 托管基建"**,而非纯 AI 全自动。
3. **最难也最值钱的一环是"测试判断"(断言 / 业务裁决)**,AI 目前显著不足;且需警惕"LLM 当裁判"导致的成绩注水。
4. **市场处于"高战略意愿、低实际落地"阶段**(75% vs 16%),信任在下降,资深工程师最谨慎——这意味着"可信度"而非"自动化程度"是采购决策的真正瓶颈。
