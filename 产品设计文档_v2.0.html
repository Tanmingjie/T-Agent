1. 产品概述

1.1 产品定位

面向测试工程师的内网 Web 自动化测试平台。输入业务测试用例（Excel / 测试管理平台），AI Agent 自动驱动浏览器执行，产出可长期维护的 pytest-bdd Playwright 测试代码。本地 LLM，私有化部署，数据不出内网。

1.2 核心价值

  • 降低自动化门槛：测试工程师用业务用例直接驱动 AI 执行，无需手写 Playwright 代码
  • 测试资产沉淀：AI 执行录制自动转换为高质量 BDD 测试代码，供长期维护
  • 内网友好：playwright-mcp stdio 模式绕过公司代理，本地 LLM 数据不出网
  • 自愈能力：界面变化后 Agent 自动重新定位元素，减少人工维护选择器的成本
  • 可信断言：预期结果转为结构化断言，由规则引擎验证，PASS/FAIL 可复现可定位

1.3 边界约束

类别内容
不做多人协作/账号体系、CI/CD 集成、TypeScript 代码生成、公网 LLM、跨浏览器（Firefox/WebKit）
暂不做（预留接口）测试管理平台 API 对接、Page Object 模式生成、并发执行、报告导出 HTML/Excel、失败断点续跑
核心约束私有化部署、本地 LLM（Qwen3 397B）、单人工具、playwright-mcp stdio、用例间串行执行

3.1 Excel 用例格式

字段说明示例
用例名称用例的唯一标识名称采购申请单提交验证
预置条件执行前需满足的条件（混合态：状态声明 + 操作步骤）已登录系统；环境已装好；设置环境变量 CONF=10
测试步骤有序操作步骤，支持多行（1. / ① / 换行三种格式）。数据写死在步骤里1. 进入采购申请页面

3.2 预置条件三分类

预置条件为混合态（既有状态声明，也有操作步骤），使用 LLM 做意图分类，不用关键词匹配（成功率低）。

类型示例处理方式
① 状态声明"已登录系统" / "环境已装好"映射到 Hook，平台自动处理（映射规则用户可维护）
② 操作步骤"设置环境变量 CONF=10"转为 Agent Given 步骤，在 Hook 之后执行
③ 模糊/低置信"准备好测试环境"界面标黄，用户手动选择：Hook / Given / 忽略
重要：分类结果首次确认后记忆到用例，下次执行跳过确认。用户可在用例详情页随时修改。

4. TestSpec（执行规格）v2 新增

4.1 定位：软计划，不是硬脚本

TestSpec 是介于"用例文本"和"Agent 执行"之间的结构化执行规格，由"预解析 + 词汇表 + Page Intelligence"共同生成。它统一了三件事：预解析输出 + 用户确认对象 + 代码生成输入。

关键设计：TestSpec 记录"动作 + 目标语义 + 数据 + 断言"，但不锁定具体 selector（selector 由执行时词汇表/自愈动态解析）。它是"执行计划"不是"执行脚本"，Agent 执行时仍可自愈、调整，保留 AI 弹性。

4.2 TestSpec 结构

case_id: "TC001"
name: "采购申请单提交验证"
base_url: "http://erp.corp.com/purchase"
given:                          # 来自预置条件②类（操作步骤）
  - action: set_env
    data: "CONF=10"
steps:
  - action: navigate
    target: "采购申请页"
  - action: fill
    target: "标题输入框"      # 词汇表解析为真实元素
    data: "测试采购单"          # 步骤里写死的数据
  - action: click
    target: "提交"              # 不锁 selector，执行时解析
assertions:                     # 来自预期结果，翻译为结构化断言
  - type: text_equals
    target: "状态字段"
    expected: "待审批"
  - type: element_visible
    target: "提交成功提示"

4.3 引入 TestSpec 的收益

① 执行更稳定

  • Agent 按已确定 Spec 执行，不"边想边做"
  • 本地 LLM 运行时推理压力降低

② 可审查可修正

  • 用户执行前看到完整 Spec，可改
  • 修正后记忆，下次复用

③ 代码生成更精确

  • Spec 已含动作 + 断言
  • 从 Spec 生成，不依赖执行录制的噪声

④ 复用与版本化

  • Spec 可保存、diff、版本管理
  • 自愈更新 selector 后回写 Spec

5. 断言机制 v2 核心强化

5.1 核心思路：把"判断"从运行时移到翻译时

问题：预期"状态变为待审批"若靠 LLM 在运行时看 A11y Tree 眼判 PASS/FAIL，不可靠——页面任何位置有"待审批"字样就误判，LLM 倾向脑补成功，两次判断可能不一致，本地模型误判率更高。
方案：预期结果 → 结构化断言（Assertion）。LLM 只在生成 TestSpec 时做一次性翻译（可审查可修正），执行时由规则引擎确定性验证。把"判断"从运行时移到翻译时。
预期结果
自然语言

一次性，可审查

存入 TestSpec

可复现可定位

5.2 五类结构化断言（按可靠性排序）

类型可靠性能力断言示例
① DOM 元素断言最高元素存在/可见/可点击/数量/状态（disabled/checked）element_visible(role=status name=待审批)
② 文本内容断言高限定具体元素内匹配文本（=/包含/正则）text_equals([data-field=status], "待审批")
③ URL/导航断言高URL 等于/包含/匹配，页面标题url_contains("/approval/pending")
④ 数据断言（Custom Tool）业务级数据库查询/内部 API 校验，最贴业务真相db_query(order_id, status=待审批)
⑤ LLM 语义断言兜底无法结构化时的最后手段，标记低置信需复核llm_judge("页面显示成功提示")
关键：文本断言限定在具体元素内匹配，而非全页搜索——这直接解决了"全页有'待审批'就误判"的问题。降级策略：能转 DOM/文本/URL 就转，转不了用 Custom Tool，最后才用 LLM 兜底。

5.3 翻译规则

预期结果（原文）翻译为 Assertion说明
"状态变为待审批"text_equals(status元素, "待审批")词汇表定位"状态"元素，限定范围匹配
"提交成功，跳转列表页"url_contains("/list") AND element_visible(成功提示)一个预期拆成多个断言（AND 关系）
"列表新增一条记录"element_count(列表行, increased) 或 db_queryUI 计数或数据库验证（后者更可靠）
"页面显示错误提示"element_visible(role=alert)语义角色定位

5.4 断言与自愈的关系

断言失败的两种归因

  • 真失败：操作执行了但业务结果不对 → FAIL（真 bug）
  • 断言 selector 失效：界面变了，断言定位元素的 selector 失效 → 触发自愈重定位断言目标

带来的提升

  • 结果可信：规则引擎判定，可复现
  • 失败可定位：明确哪个断言、期望/实际、附证据截图
  • 代码精确：Assertion 直接映射 Playwright expect()
设计要点：Healing Subagent 同时服务于"操作元素定位"和"断言元素定位"。断言失败也可触发自愈。

6. Page Intelligence 层

6.1 设计目标

解决业务用例文本质量差的核心问题：用例里的模糊词（"提交"）映射到页面真实 UI 元素（"保存并提交"按钮）。同时辅助 Self-Healing 快速重定位变化后的元素，辅助断言翻译。

6.2 三个核心能力

能力说明可见可维护
词汇表 Vocabulary业务词→UI元素映射。执行前注入 Prompt，自愈/断言翻译时查表可新增/修改/废弃，手动条目优先级高于 AI 扫描
操作图谱 Action Map页面按功能区分组的可交互元素，含角色/名称/状态可标注业务含义，标记废弃，显示 stale
用例预解析执行前用 LLM + 词汇表生成 TestSpec（含断言翻译）默认手动确认（可配自动），歧义词高亮

6.3 扫描策略：A+C 组合

  • 策略 A（手动首次扫描）：用户手动触发，选择 Session Profile（登录态），扫描 base_url 及关键子页面，建立初始词汇表
  • 策略 C（执行中自动增量）：Agent 执行过程中访问到新页面时自动增量扫描，补充词汇表
冷启动处理：首次对某 base_url 执行用例时，若词汇表为空，提示先执行一次 Page Scan，保证 TestSpec 生成有基础数据。

6.4 缓存设计

字段说明
缓存 Keyurl_pattern + page_title + login_role（组合唯一）
URL 路由匹配/order/123 和 /order/456 共用一条，Key 为 /order/{id}
更新触发手动 / 自愈失败标记 stale / 定时（可配置）
登录态隔离同一 URL 不同角色分开存储词汇表

7. Agent Harness 层

7.1 Session Profile（登录态管理）

将账号信息、登录脚本、Cookie 缓存打包为可复用配置单元。Suite 绑定 Profile，Page Scanner 和 Hook 共用同一套机制。

字段说明
nameProfile 名称，如"采购员账号"
login_aw登录 AW 引用（用户已有 pytest 脚本）
cookie_storeCookie 持久化路径
valid_untilCookie 有效期，过期自动重新执行 login_aw
base_url该 Profile 对应的系统入口
复用机制：before_case Hook 检查 valid_until，未过期直接注入 Cookie 跳过登录；过期则执行 login_aw 刷新。Cookie 有效期内跨用例共享，同 Suite 多用例无需重复登录。

7.2 ReAct 执行模式

步骤说明录制字段
Reason读 StepPlan + A11y Tree(截断) → 输出思考链，决策工具和元素step.reasoning
Act执行 MCP tool / Custom Tool / Hook 动作tool_name / tool_input / intent
Observe获取 A11y Tree + 截图 + URL，更新 StepPlantool_result / screenshot / url

StepPlan（TodoWrite）：TestSpec.steps → StepPlan 列表，状态机 pending/active/done/failed/skipped。每步完成 LLM 调用 mark_step_done 更新。防止多步骤用例跳步/遗漏/重复。

7.3 Self-Healing（自愈机制）

触发条件：元素未找到、操作无效（页面无变化）、连续重复（循环检测）、超时、断言 selector 失效。

多策略定位优先级

优先级策略原理 / 抗变化能力
P1 最高语义角色定位role + accessible name，由内容决定，不随样式变化 — 极强
P2文本内容定位元素可见文字，文案不改即有效 — 强
P3语义属性定位aria-label / placeholder / title — 中强
P4位置关系定位相对稳定锚点的位置关系 — 中
P5 最低截图视觉定位截图 → LLM 视觉分析，兜底 — 弱

自愈流程

  1. 失败触发 → 抓取当前 A11y Tree 全量快照 + 截图（双通道，借鉴 browser-use）
  2. LLM 分析：失败信息 + 当前快照 + 原始意图 → 生成 3 个候选方案（含置信度）
  3. 按 P1→P5 逐一尝试，成功则继续并将新选择器回写 ActionStep + 词汇表
  4. 全部失败 → 重新规划绕路 → 超出 max_heal_attempts → FAIL + 自愈报告
核心原则：ActionStep 存"操作意图"（自然语言），不只存选择器。意图是自愈重定位的依据；选择器是实现细节，界面变了就废了。词汇表是第一优先查询来源。

7.4 Subagent（上下文隔离）

避免上下文污染，按用例级隔离（不是步骤级——用例内步骤有状态依赖必须共享 context）。

Subagent 类型作用优先级
用例 Subagent每用例独立 context，批量执行时互不污染。用例A失败的错误上下文不影响用例BP0 必做
自愈 Subagent自愈的失败尝试/快照/截图在独立 context 分析，主 context 只拿回一行摘要P1 强烈建议
Scanner SubagentPage Intelligence 扫描独立 context，大量 A11y 数据不污染执行P1 自然实现
边界：隔离粒度是用例级。用例内步骤之间不拆 Subagent（步骤2需要知道步骤1做了什么，状态必须同 context 传递）。

7.5 Context Compact（上下文压缩）

  • L1 步骤归档：StepPlan 标记 done 后，从活跃 context 移出，归档为一行摘要
  • L2 A11y Tree 截断：注入前按当前步骤关键词做相关度过滤，只保留相关元素（借鉴 browser-use）
  • L3 Skill 卸载：离开页面时卸载对应 PageSkill
  • 自愈特殊处理：自愈中不压缩（需完整失败上下文），自愈成功后整个过程压缩为一行立即归档

7.6 System Prompt 分层 + Skill 体系

层级内容来源
Base角色定义、ReAct 格式、Behavioral Nudges（防循环指令）、TEST_RESULT 规范、安全边界平台固定
Context用户自定义业务知识，支持 {{base_url}} 变量插值Case > Suite > 全局
TaskTestSpec、StepPlan、Hook 状态每次执行注入
ToolsMCP + 自定义工具描述（相关度截断）动态生成

三种 Skill：DomainSkill（Suite 级业务术语，借鉴 SKILL.md）、PageSkill（按 URL 动态加载/卸载）、ToolSkill（相关度过滤注入）。

7.7 Hooks 系统

Hook 时机典型用途失败处理
before_suite整批初始化全局环境终止整批
before_case ★init_env_aw + LoginHook当前用例 FAIL，不进 Agent
after_case ★清理数据，退出登录记录错误，不影响结果
on_heal自愈触发时通知上报仅通知
on_failure失败时截图 + 告警仅通知
after_suite生成汇总报告记录错误

7.8 Custom Tools（自定义工具）

用户已有自动化脚本可接入，LLM 按需调用（区别于 Hook 的强制执行）。

接入方式

  • Python 装饰器 @tool（借鉴 browser-use @tools.action）
  • YAML 配置（command 字段，无需改原脚本）

典型场景

  • 数据库断言（数据断言的实现）
  • 内部 API 校验、测试数据准备
  • 发送钉钉/企微通知
Hook vs Tool：Hook = 每次必须执行、与测试意图无关的前置/后置（初始化、登录）。Tool = LLM 按任务上下文决定调用的能力（数据断言、API 校验）。

7.9 Permission（操作许可）

  • 高危操作白名单：tool_call 含"删除/提交/支付/确认"等词 → 暂停确认
  • 环境保护锁：base_url 命中 prod 域名 → 必须用户显式解锁
  • 触发位置：Reason 之后 Act 之前（步骤 8），asyncio.Event 暂停，UI 弹窗
  • 信任模式：可配置跳过所有拦截

8.2 BDD 代码生成规则

BDD 结构来源说明

WhenTestSpec.steps（业务步骤粒度，非 tool_call 粒度）按原始用例步骤合并
ThenTestSpec.assertions断言直接映射 expect()
  • 选择器：词汇表提供稳定选择器，优先 get_by_role / get_by_label
  • 代码质量：可读性 > 简洁性，业务含义中文注释，供长期维护
  • 验证：ast.parse 语法验证 + black 格式化 +（建议）自动 pytest 回放验证"可信代码"

12. 风险与约束

风险描述缓解措施
本地 LLM tool_call 稳定性Qwen3 tool_call 格式可能偶发错误Harness 层格式容错 + 重解析；快速失败提示切换模型（LiteLLM 零成本）
词汇表冷启动首次无词汇表，TestSpec/断言翻译无依据A+C 策略：首次提示先扫描 base_url 建立基础词汇表
断言翻译准确率LLM 翻译预期为断言可能出错翻译结果在 TestSpec 审查界面可见可改；降级策略；用户修正后记忆
预置条件分类准确率LLM 分类模糊预置条件可能出错置信度阈值 + 标黄 + 用户确认兜底；结果记忆
生成代码可执行性ast 语法对 ≠ 能跑通生成后自动 pytest 回放验证，通过才标"可信代码"
并发与资源多 Subagent 打同一 Ollama 实例阶段一/二明确串行执行用例，并发作为后期优化
