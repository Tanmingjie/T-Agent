> 阅读顺序:先读「0. 给实现者的话」→「1. 产品本质」→「2. 架构」→「6. 实施计划」,然后从阶段一 T-01 开始。

## 0. 给实现者的话

你将实现一个**内网 Web 自动化测试平台**。核心链路是:

```
业务测试用例(Excel) → 生成 TestSpec(结构化执行规格+断言) → AI Agent 驱动浏览器执行 → 结构化断言验证 → 产出可维护的 pytest-bdd Playwright 代码
```

**实现原则(务必遵守):**

1. **前后端彻底分离**:所有功能通过 HTTP API 暴露,前端只调 API。即使现阶段是本地单机运行,也当成"前端 + 后端服务"两个独立部分。这是为未来多人化预留的。
2. **数据层抽象**:用 SQLModel,业务代码不直接写 SQL。未来 SQLite→PostgreSQL 只改连接串。
3. **输入/输出抽象**:Excel 解析器与未来的"用例管理平台拉取器"都产出同一个 `TestCase` 结构体;执行结果都落在 `ExecutionRecord`。下游只认这两个结构,不关心来源/去向。
4. **数据预留同步字段**:核心表预留 `updated_at` / `owner` / `external_id`,为未来对接用例管理平台预留。
5. **分阶段、可验证**:严格按实施计划的阶段推进。每个阶段有明确验收标准,跑通了再进下一阶段。不要跳阶段、不要过度设计未来阶段的东西。

**运行环境(本地):**
- 浏览器层:`playwright-mcp`(stdio 模式),已验证可绕过内网代理。**必须用 stdio,不要用 CDP HTTP 连接**(内网代理会拦截导致 504)。
- LLM:本地 Qwen3 397B,通过 Ollama,用 LiteLLM 接入。模型已部署,无需关心部署细节。
- 浏览器进程:用户本机的 Chrome,能访问被测内网系统(网络/代理/登录态都是本机的)。

**不要做的事(明确排除):**
- 不做多人协作/账号体系/CI-CD 集成/TypeScript 代码生成/跨浏览器
- 用例间**串行执行**,不做并发
- 用例管理平台集成**留到最后**(阶段四之后),现在只预留扩展点
- 测试数据**写死在用例步骤里**,不做数据工厂/动态生成

## 1. 产品本质

| 项 | 内容 |
|---|---|
| 定位 | 内网 Web 业务测试自动化执行平台,面向测试工程师 |
| 输入 | 业务测试用例(Excel,字段:用例名称/预置条件/测试步骤/预期结果) |
| 输出 | 测试执行结果 + pytest-bdd Playwright 测试代码(作为长期测试资产) |
| 核心难点 | ① 用例文本质量差(模糊词需映射真实UI) ② 断言必须可信(不能LLM眼判) ③ 界面变化自愈 ④ 本地LLM能力有限 |

### 为什么是这套技术方案(背景,帮助你理解决策)

- 放弃 browser-use:它用 CDP HTTP 连接,被公司内网代理拦截(504),无法使用。
- 选 playwright-mcp stdio:stdio 进程间通信不走 HTTP,绕过代理。A11y Tree 直接映射 Playwright `get_by_role`,利于代码生成。
- 自建 Agent Harness:在 MCP 之上自己实现 ReAct 循环、自愈、断言、上下文管理等,这是核心价值所在。

## 2. 整体架构

### 五层

```
① 输入层          Excel → TestCase + 预置条件三分类
② TestSpec 层     结构化执行规格 + 断言翻译(本产品关键中间产物)
③ Page Intelligence  词汇表 / 操作图谱 / 用例预解析(解决用例文本质量差)
④ Agent Harness   ReAct + 自愈 + 断言验证(执行引擎,核心)
⑤ 输出层          BDD 代码生成 + 执行记录
                  ┕━ 之上:工程化界面(FastAPI + React)
```

### 部署形态(重要)

- **现在(单人)**:整个产品(FastAPI 后端 + React 前端 + Agent + Chrome + SQLite)全部跑在用户本机。React 访问 `localhost`。这就是混合架构的"单机退化版",未来不返工。
- **未来(多人)**:本地仍是完整应用,执行引擎不动。本地**主动**通过 HTTP 同步数据到中心(可能复用已有的用例管理平台)——本地→中心是出站请求,防火墙放行。绝不让中心反向连本地。

### 完整执行流(16步)

| # | 动作 | 涉及机制 |
|---|---|---|
| 1 | Excel 导入 → TestCase | 输入层 |
| 2 | 预置条件三分类(LLM) | 状态声明→Hook / 操作→Given / 模糊→用户确认 |
| 3 | 用例预解析 + 断言翻译 → 生成 TestSpec | Page Intelligence + 断言翻译 |
| 4 | 启动 TestCase Subagent | Subagent 隔离(每用例独立 context) |
| 5 | before_case Hooks | init_env_aw → LoginHook(Cookie 复用) |
| 6 | Prompt 组装 + Skill 注入 | Base+DomainSkill+Task(TestSpec)+ToolSkill |
| 7 | Reason(LLM 推理) | ReAct;读 StepPlan + A11y Tree(截断) |
| 8 | Permission 检查 | 高危词/prod 环境 → 暂停确认 |
| 9 | Act(执行工具) | ReAct;MCP/Custom Tool;录制 ActionStep |
| 10 | Observe(感知) | ReAct;A11y + URL + 截图;PageSkill 切换 |
| 11 | 断言验证 | 规则引擎验证(非 LLM 眼判) |
| 12 | Context Compact + 循环检测 | 步骤归档/A11y截断/Skill卸载 |
| 13 | Self-Healing(失败时) | Healing Subagent;P1→P5+截图+词汇表 |
| 14 | after_case Hooks | 清理/退出登录/告警 |
| 15 | CodeGenerator → BDD 代码 | TestSpec+录制 → feature+step_defs |

## 4. 核心数据结构

> 全部用 SQLModel(继承 SQLModel,table=True 的做持久化;纯结构用 pydantic BaseModel)。核心表预留 `updated_at` / `owner` / `external_id`。

```python
# input/models.py

class TestCase(BaseModel):
    id: str
    name: str
    preconditions: list[str]      # 已拆分的预置条件
    steps: list[str]              # 已拆分的测试步骤(数据写死在文本里)
    expected: list[str]           # 预期结果
    base_url: str
    suite_id: str | None = None
    external_id: str | None = None  # 预留:对应用例管理平台的用例ID

class Assertion(BaseModel):
    type: str          # element_visible | text_equals | url_contains | element_count | custom_tool | llm_judge
    target: str        # 目标语义描述(执行时解析为 selector)
    selector: str | None = None  # 可选,词汇表解析后填入
    expected: str | None = None
    confidence: str = "high"      # high | low(llm_judge 为 low)

class SpecStep(BaseModel):
    action: str        # navigate | fill | click | select | hover | wait | ...
    target: str        # 目标语义描述,不锁 selector
    data: str | None = None       # 步骤里写死的数据
    expect: list[Assertion] = []  # 该步骤的即时断言(可选)

class TestSpec(BaseModel):
    case_id: str
    name: str
    base_url: str
    given: list[SpecStep] = []     # 来自预置条件②类(操作步骤)
    steps: list[SpecStep]          # 来自测试步骤
    assertions: list[Assertion]    # 用例级最终断言(来自预期结果)

class PreconditionItem(BaseModel):
    text: str
    type: str          # state_hook | action_step | ambiguous
    hook_ref: str | None = None    # 状态声明映射到的 Hook
    confidence: float
    confirmed_by_user: bool = False

class ActionStep(BaseModel):
    step_no: int
    tool_name: str
    tool_input: dict
    reasoning: str = ""            # ReAct 思考链
    intent: str = ""              # 操作意图(自愈重定位的依据)
    tool_result: str = ""
    screenshot: str | None = None  # 文件路径
    url: str = ""
    heal_attempts: list[dict] = []
    assertion_results: list[dict] = []
    is_custom_tool: bool = False
    is_hook_action: bool = False
    duration_ms: int = 0

class ExecutionRecord(BaseModel):
    exec_id: str
    case_id: str
    suite_id: str | None = None
    steps: list[ActionStep] = []
    passed: bool = False
    final_result: str = ""
    generated_code: str = ""
    token_usage: int = 0
    heal_count: int = 0
    start_time: float
    end_time: float | None = None

class SessionProfile(BaseModel):
    name: str
    login_aw: str          # 登录 AW 引用(用户已有 pytest 脚本)
    cookie_store: str      # Cookie 持久化路径
    valid_until: float | None = None
    base_url: str

class Suite(BaseModel):
    id: str
    name: str
    base_url: str
    session_profile: str | None = None
    page_intelligence_id: str | None = None
    code_generator: str = "BDDGenerator"
    custom_prompt: str = ""
    hooks: dict = {}       # {"before_case": [...], "after_case": [...]}

class PageVocabulary(BaseModel):
    url_pattern: str       # /order/{id}
    page_title: str
    login_role: str
    vocabulary: dict        # {业务词: {role, name, confidence}}
    action_map: list = []
    stale: bool = False
    scanned_at: float

class ToolDef(BaseModel):
    name: str
    description: str
    parameters: dict        # JSON Schema
    returns: str
    when_to_use: str
    timeout_seconds: int = 30
```

## 5. 各模块详细规格

### 5.1 输入层

**Excel 解析(`input/excel_parser.py`)**
- openpyxl 读取。字段:用例名称 / 预置条件 / 测试步骤 / 预期结果。
- 步骤拆分:支持 `1. 2. 3.` / `① ② ③` / 纯换行 三种格式。
- 容错:空行跳过、合并单元格处理。
- 输出 `list[TestCase]`。

**预置条件三分类(`harness/precondition.py`)**
- 用 LLM 做意图分类(**不用关键词匹配**,成功率低)。三类:
  - `state_hook`(状态声明,如"已登录系统""环境已装好")→ 映射到 Hook
  - `action_step`(操作步骤,如"设置环境变量 CONF=10")→ 转为 TestSpec.given
  - `ambiguous`(模糊/低置信)→ 标记待用户确认
- 输出 `list[PreconditionItem]` 含置信度。
- 分类结果记忆到用例(持久化),下次跳过。
- 状态声明→Hook 的映射规则用户可维护(关键词→Hook 名的映射表)。

### 5.2 TestSpec 生成(`intelligence/pre_analysis.py`)

- **定位:软计划,不是硬脚本。** 记录"动作+目标语义+数据+断言",**不锁定具体 selector**(执行时词汇表/自愈动态解析)。
- 输入:TestCase + 预置条件分类结果 + 词汇表(阶段一可无)
- 处理:
  - 测试步骤 → SpecStep(action + target 语义 + data)
  - 模糊词用词汇表增强("提交"→target="保存并提交")。阶段一无词汇表时纯 LLM 解析。
  - 预期结果 → Assertion(见 5.3 断言翻译)
  - 预置条件 action_step → TestSpec.given
- 输出 TestSpec。用户可在执行前审查/修改(默认手动确认,可配自动)。

### 5.3 断言机制(`harness/assertion.py`)★ 核心

**核心思想:把"判断"从运行时移到翻译时。**
- LLM 只在生成 TestSpec 时做**一次性翻译**(预期结果 → 结构化 Assertion),可审查可修正。
- 执行时由**规则引擎确定性验证**,不靠 LLM 眼判。

**五类断言(按可靠性,降级使用):**
1. `element_visible` / `element_count` — DOM 元素断言(最可靠)
2. `text_equals` / `text_contains` — **限定具体元素内**匹配(不是全页搜!解决"全页有'待审批'就误判")
3. `url_contains` / `url_equals` — URL/导航断言
4. `custom_tool` — 数据断言(数据库查询/API,通过 Custom Tool,最贴业务真相)
5. `llm_judge` — LLM 语义断言(兜底,标记 confidence=low,需人工复核)

**翻译规则示例:**
| 预期(原文) | Assertion |
|---|---|
| "状态变为待审批" | text_equals(status元素, "待审批") |
| "提交成功,跳转列表页" | url_contains("/list") AND element_visible(成功提示) — 一对多拆解 |
| "页面显示错误提示" | element_visible(role=alert) |

**降级策略:** 能转 DOM/文本/URL 就转,转不了用 custom_tool,最后才 llm_judge。

**断言失败的两种归因:**
- 真失败:操作执行了但业务结果不对 → FAIL
- 断言 selector 失效:界面变了 → 触发自愈重定位断言目标(Healing 同时服务操作定位和断言定位)

### 5.4 Agent Harness

**ReAct Loop(`harness/react_loop.py`)**
- Reason → Act → Observe 循环。
- Reason:读 StepPlan 状态 + A11y Tree(截断) + PageSkill → 决策。记录 reasoning。
- Act:执行 MCP tool / Custom Tool / Hook 动作。记录 intent。
- Observe:A11y Tree + 截图 + URL,更新 StepPlan。
- 循环检测:连续 3 步相同 tool_call → 终止。
- max_steps 上限。解析 LLM 输出的 `TEST_RESULT: PASS/FAIL`(但最终以断言结果为准)。

**StepPlan / TodoWrite(`harness/step_plan.py`)**
- TestSpec.steps → StepPlan 列表,状态机:pending/active/done/failed/skipped。
- LLM 每步完成调用 `mark_step_done(step_no)` 工具更新状态。
- 防止多步骤用例跳步/遗漏/重复。序列化为 System Prompt 片段。

**Self-Healing(`harness/healing.py`)— Healing Subagent**
- 独立 context(不污染主 Agent)。
- 触发:元素未找到 / 操作无效 / 连续重复 / 超时 / 断言 selector 失效。
- 输入:失败信息 + 当前 A11y Tree + 截图(双通道) + intent + 词汇表。
- 多策略定位优先级:P1 语义角色(role+name) → P2 文本 → P3 属性(aria-label) → P4 位置关系 → P5 截图视觉。
- 生成 3 个候选方案(含置信度),按 P1→P5 尝试。
- 成功:回写 ActionStep + 词汇表。主 context 只记一行摘要。
- 失败:重新规划 → 超 max_heal_attempts → FAIL。
- **关键原则:ActionStep 存"操作意图"而非只存选择器。词汇表第一优先查询。**

**Subagent 隔离(`harness/agent.py` + `orchestrator.py`)**
- 用例级隔离(非步骤级——用例内步骤有状态依赖,必须共享 context)。
- 每用例 = 独立 async 协程,独立 messages list。
- 用例A失败的错误上下文不污染用例B。
- 自愈 Subagent、Scanner Subagent 同理独立 context。
- 实现:asyncio.create_task,各自维护 messages,不用额外框架。

**Context Compact(`harness/context.py`)**
- L1:done 步骤从活跃 context 移出,归档为一行摘要。
- L2:A11y Tree 注入前按当前步骤关键词相关度截断(借鉴 browser-use)。
- L3:离开页面卸载 PageSkill。
- 自愈中不压缩(需完整失败上下文),自愈成功后整个过程压缩为一行立即归档。
- token 用量监控,接近上限触发。

**System Prompt 分层(`harness/prompt.py`)**
- Base(平台固定:角色/ReAct格式/Behavioral Nudges防循环/TEST_RESULT规范/安全边界)
- Context(用户自定义业务知识,{{变量}}插值,优先级 Case>Suite>全局)
- Task(TestSpec/StepPlan/Hook状态)
- Tools(MCP+自定义工具,相关度截断)
- **Behavioral Nudges 对本地 LLM 尤其重要**(如"如果已点击同一按钮2次,停下重新分析")。

**Skill 体系(`harness/skills.py`)**
- DomainSkill:Suite 级业务术语,执行前注入。
- PageSkill:按 URL 动态加载/卸载。
- ToolSkill:相关度过滤注入。

**Hooks(`harness/hooks.py`)**
- before_suite / before_case★ / after_case★ / on_heal / on_failure / after_suite。
- before_case 失败 → 用例 FAIL,不进 Agent。
- 顺序执行队列,execution_context 共享状态传递。

**Session Profile(`harness/session.py`)**
- 账号 + login_aw + Cookie 缓存。
- before_case 检查 valid_until:未过期注入 Cookie 跳过登录;过期执行 login_aw 刷新。
- Cookie 有效期内跨用例共享。Suite 绑定 Profile。

**Custom Tools(`harness/tools.py`)**
- `@tool` 装饰器注册(借鉴 browser-use @tools.action 风格)。
- 也支持 YAML 配置接入(command 字段)。
- LLM **按需调用**(区别于 Hook 强制执行)。
- 数据断言通过 Custom Tool 实现(数据库查询)。

**Permission(`harness/permission.py`)**
- 高危词白名单(删除/提交/支付/确认)+ 环境锁(prod URL)。
- 触发位置:Reason 后 Act 前。
- asyncio.Event 暂停 + 等待确认。可配信任模式(跳过)。

**LLM 接入(`harness/llm.py`)**
- LiteLLM 封装,连本地 Ollama(Qwen3 397B)。
- **tool_call 格式容错**:本地模型可能 tool_call 格式出错,需 JSON 解析失败重试/重解析。
- token 用量统计接口。
- 模型接口抽象,可切换(快速失败后换模型)。

**MCP 客户端(`mcp/client.py`)**
- 用**官方 mcp Python SDK**(StdioServerParameters)。
- 连 playwright-mcp(`npx @playwright/mcp`)。
- 封装 list_tools() / call_tool()。
- 工具格式 MCP → LiteLLM tool 格式转换。
- **必须 stdio,不要 CDP HTTP**(绕过内网代理)。

### 5.5 Page Intelligence(阶段三)

- Scanner Subagent:登录(Session Profile)→ 扫描 → LLM 提炼词汇表 → 写 DB。
- 扫描策略 A+C:A(手动首次扫描 base_url)+ C(执行中增量补充)。冷启动:首次词汇表为空时提示先扫描。
- 词汇表:业务词→UI元素映射。用户可见可维护,手动条目优先级高于 AI 扫描。
- 缓存 Key:url_pattern + page_title + login_role。路由匹配(/order/{id})。
- 更新触发:手动 / 自愈失败标记 stale / 定时。

### 5.6 输出层(阶段三)

**CodeGenerator(`codegen/base.py` + `bdd.py`)**
- 抽象接口 `generate(record: ExecutionRecord) → str`,Suite 配置时选择实现。
- BDDGenerator(默认):TestSpec + 录制 → feature(Gherkin) + step_defs.py + conftest.py。
  - Given ← TestSpec.given + Hook 状态
  - When ← TestSpec.steps(业务步骤粒度,**不是 tool_call 粒度**)
  - Then ← TestSpec.assertions(**断言直接映射 Playwright expect()**)
  - 选择器优先 get_by_role / get_by_label
  - 可读性 > 简洁性,中文业务注释
- 验证:ast.parse 语法 + black 格式化 +(建议)自动 pytest 回放验证"可信代码"。

| MCP 客户端 | 官方 mcp Python SDK | StdioServerParameters,锁版本 |

## 8. 风险与应对

| 风险 | 应对 |
|---|---|
| 本地 LLM tool_call 不稳定 | Harness 层格式容错+重解析;快速失败提示切模型 |
| 词汇表冷启动 | 阶段一 TestSpec 纯 LLM 生成;阶段三引入词汇表后增强;首次提示先扫描 |
| 断言翻译出错 | 翻译结果在 TestSpec 审查界面可改;降级策略;记忆 |
| 预置条件分类出错 | 置信度阈值+标黄+用户确认兜底 |
| 生成代码语法对≠能跑通 | 自动 pytest 回放验证才标"可信" |
| 多 Subagent 抢 LLM 资源 | 阶段一/二明确串行执行 |
