# T-Agent 平台化设计草案(Web 多租户版)

> 2026-06-10 头脑风暴产出(superpowers brainstorming 流程:逐题确认决策 → 沉淀本文)。
> 定位:从单机版演进为**全公司多产品线共享的测试自动化平台**。
> 本文是方向性设计,实施前各里程碑仍需细化为规格(对齐《实现规格说明书.md》的写法)。

## 0. 已拍板的决策(用户确认)

| 决策点 | 选择 |
|---|---|
| 规模 | **大**:全公司平台、产品线众多 |
| 部署 | **公司内部 K8s/容器平台**(API/worker 各自镜像,worker 内置 Chromium,可扩缩容) |
| Custom Tool 安全模型 | **只允许 HTTP 请求型工具**(不跑 shell;查 DB 等场景引导项目方自包 HTTP 接口) |
| Suite 协作 | **项目内全开放**(成员都可见可跑;改/删限 owner + 项目管理员) |
| 版本模型 | **Suite 也绑版本**(层级:项目→版本→Suite→Run;新版本从上一版本**显式拷贝/继承** Suite) |
| 词汇表归属 | **项目级、跨版本共享**(项目内按 base_url 区分系统/环境;页面变更靠 stale 重扫) |
| 角色 | **三角色+自助开通**:平台管理员 / 项目管理员 / 测试人员;申请建项目→平台审批 |
| MVP 范围 | **多租户核心链路**(项目/版本/角色 + 项目级 LLM + Postgres + API/worker 分离;配置界面化与 IDaaS 放二期) |
| 数据库 | **Postgres**(JSONB 索引查 spec/断言、LISTEN/NOTIFY 省 Redis、事务性 DDL 利迁移、asyncpg 成熟;MySQL 仅当公司托管运维只支持它时反选) |
| 开发顺序 | **本地优先,K8s 延后**(环境准备周期长;本地即按 API/worker 双进程形态开发,进程边界=未来 Pod 边界) |

## 1. 租户模型

```
平台
└── 项目 (Project)            ← 租户边界 / 配额边界 / 配置边界
    ├── 项目级配置:LLM、HTTP Tools、Skills、Hooks/SessionProfile、词汇表(按 base_url)
    ├── 成员(角色:项目管理员 / 测试人员)
    └── 版本 (Version)
        └── Suite(owner=创建人,项目内全开放)
            └── Run → Case 结果 / 断言 / 截图 / 生成代码
```

- **版本继承**:新建版本时可选「从版本 X 拷贝 Suite」,显式动作、拷贝后独立演进(修用例只改当前版本)。
- **词汇表不随版本拷贝**:项目级资产,base_url 维度已实现(2026-06-10);改版靠自愈 `mark_stale` + 重扫。
- 铁律5 预留的 `updated_at` / `owner` / `external_id` 在此启用;所有核心表加 `project_id`(Suite/Run 再加 `version_id`)。

## 2. 配置:从进程级 env → 项目级 DB 实体

单机版最深的假设是「一进程一套配置」。平台版全部实体化:

| 配置 | 单机版 | 平台版 |
|---|---|---|
| LLM | `.env`(进程级) | `ProjectLLMConfig` 表:model/api_base/api_key(**加密落库**,界面只显尾号)+ 连通性自检按钮(复用 `--check-llm` 逻辑)。每 run 按项目实例化 `LLMClient` |
| Custom Tools | env 指向本地 YAML,跑 shell | `ProjectHttpTool` 表:URL/method/headers/body 模板参数化;执行=受控 HTTP 调用(超时/响应大小限制/**仅限内网网段的出站策略**)。`custom_tool` 断言语义不变 |
| Skills | 内置 + `Suite.custom_prompt` | 内置 DomainSkill 平台共享;`ProjectSkill` 表(项目业务常识);suite 级 custom_prompt 保留 |
| Hooks/Session | `Suite.session_profile` 名称引用本地文件 Cookie | SessionProfile 实体加 project 维度;**Cookie=用户凭据,加密落库**、严格项目隔离 |
| 词汇表 | 全局表(base_url 作用域) | 加 `project_id`;界面维护已有(VocabularyPage),套上项目上下文即可 |

执行链**不需要大改**:`make_agent` 本就接收 hooks/tools_registry/skills/vocab_resolver 注入,改造点集中在「配置从项目实体读取」。唯一例外:`LLMClient` 要从进程级改为 per-run 构造。

## 3. 执行架构:API 与 worker 分离(K8s)

```
浏览器 ──HTTPS──> API 服务(Deployment, 多副本)
                    │  入队(Postgres 任务表: run_queue)
                    ▼
              Worker 服务(Deployment, 内置 Chromium + playwright-mcp)
                    │  领任务(SELECT ... FOR UPDATE SKIP LOCKED)
                    │  每 case 起独立 mcp/浏览器(现有 orchestrator/parallelism 平移)
                    ▼
              Postgres(结果) + 对象存储/PVC(截图、generated code)
                    │
              SSE/进度:worker 写进度表或 Redis pub/sub → API 转 SSE
```

- **队列选型**:首选 **Postgres 任务表 + SKIP LOCKED**(少一个组件,公司平台量级足够;真到瓶颈再换 Redis/专用队列,接口抽象好即可)。这延续了项目「数据层抽象、存储可替换」的原则。
- **worker = 现有 `execution_worker.spawn_run` 的出进程化**:独立进程 + 独立 loop + 独立 Store 的隔离模式已经验证过,平移到独立 Deployment 是结构平移不是重写。
- **SSE 桥接**:单机版靠 `call_soon_threadsafe` 跨线程;跨进程后改为 worker 写进度(表轮询或 Redis pub/sub),API 端长连接转发。事件协议(`suite_start/case_start/phase/spec_ready/step_change/case_result/suite_done`)不变。
- **权限审批**(permission.py 高危拦截):单机版 `threading.Event` → 平台版「审批工单」表:worker 挂起轮询,API 通知项目内有权者,**超时默认拒绝**(与现状一致)。
- **配额**:项目级最大并发 run / 浏览器实例数;worker HPA 按队列深度扩缩。
- **产物**:截图/代码从本地目录 → 对象存储(或 PVC),路径按 `project/run` 分桶;`ExecutionRecord` 存引用。

## 4. 安全要点

1. **不执行用户 shell**(已拍板 HTTP 型工具)——平台化最大风险直接消除。
2. HTTP 工具出站限制:仅内网网段、禁元数据地址(169.254.x)、限重定向,防 SSRF 打到 K8s/云控制面。
3. 加密落库三件套:LLM api_key、SessionProfile Cookie、HTTP 工具里的 header 凭据。统一一个 `EncryptedStr` 字段类型 + KMS/密钥管理。
4. `browser_run_code_unsafe`(Cookie 注入/抓取用)只在 worker 沙箱内对被测站点执行,平台侧无暴露面。
5. 审计日志:配置变更(谁改了 LLM/Tool)、执行触发、审批动作——全公司平台必备。

## 5. 角色与 IDaaS

- 三角色:**平台管理员**(审批建项目、平台配额、全局运维)/ **项目管理员**(LLM、Tools、Skills、Session、成员、版本)/ **测试人员**(Suite、执行、词汇表维护)。
- 一期:简单本地账号(或网关 header 透传用户名),**鉴权层做成接口**(`AuthProvider`),二期换 IDaaS(OIDC/OAuth2)只换实现。
- IDaaS 只解决「你是谁」;项目内角色平台自管(不绑 IDaaS 组织架构,项目≠部门)。

## 6. 底层共享清单(回答「除了 LLM/suite data 还有什么」)

**平台共享(一份)**:
- Postgres、任务队列、worker/浏览器资源池(最贵的共享资源,配额调度的对象)
- 产物存储(截图/代码)
- Agent 框架:ReAct / 断言引擎 / 自愈 / Context Compact / codegen 模板 / 内置 DomainSkills
- 用户体系 + RBAC、审计日志、SSE 通道、调度器(定时扫描这条 TODO 天然归平台调度器)

**项目隔离(机制共享、数据隔离)**:
- LLM 配置(密钥)、HTTP Tools、Skills、SessionProfile/Cookie、词汇表、Suite/版本/Run 数据

**容易漏的共享候选**:
- 预置条件分类 memory / 自愈经验:目前在进程内,平台版可沉淀为项目级(甚至平台级)跨 run 资产
- LLM token 用量统计:LLM 虽是项目自带,平台仍按项目记账(帮他们申请资源、也防滥用)
- 词汇表跨项目复用:多项目测同一内网系统时词汇表可"发布到公共库"——**暂不做**,场景出现再说(已拍板项目级)

## 7. 里程碑(2026-06-10 修订:环境准备周期长,**本地优先开发,K8s 延后**)

> 核心原则:**进程边界 = 未来的 Pod 边界**。本地就把"多进程形态"做对(API 与 worker
> 现在就拆成两个进程、走 Postgres 任务表通信),后面上 K8s 只是打包装箱,不再动架构。
> 不许因"本地开发"退回 API 进程内起线程的形态,否则二期要重拆一遍。

**M1 多租户核心链路(MVP,本地多进程形态)**
- Project/Version/角色(本地账号)、核心表加 project_id/version_id/owner
- 项目级 LLM 配置(加密)+ 界面
- SQLite → Postgres(本地跑:Windows zip 版解压即用,`initdb`+`pg_ctl` 免管理员/免 Docker;
  Repository 抽象层就位,主要是迁移与方言验证;WAL 等 SQLite 特化逻辑退场)
- **API/worker 拆成两个本地进程**(`scripts/serve.py` + 新增 `scripts/worker.py`),
  Postgres 任务表队列(SKIP LOCKED)+ LISTEN/NOTIFY 进度桥接 SSE
- 产物存储走路径抽象(本地=目录;后续对象存储/PVC 只换实现)
- 现有执行/断言/自愈/报告能力原样跑通(saucedemo live 冒烟在多进程形态下重验)

**M2 配置界面化**
- HTTP Tools / Skills / SessionProfile 界面化(词汇表界面已有,套项目上下文)
- 版本继承拷贝 Suite、版本维度报告/进度视图
- 配额与审计

**M3 K8s 部署(环境就绪后)**
- API/worker 各自镜像(worker 内置 Chromium + playwright-mcp),HPA 按队列深度扩缩
- ingress SSE 长连接验证、产物切对象存储/PVC、托管 Postgres 迁移
- 多副本下的会话/审批/调度专项验证;saucedemo 冒烟在集群形态重验

**M4 IDaaS 接入 + 平台运营**
- OIDC 对接(换 AuthProvider 实现)、自助开通审批流
- token 记账、平台管理面板
- (候选)词汇表公共库、定时扫描调度

## 8. 风险与开放问题

- **Postgres 迁移面**:Repository 抽象在,但 `storage/db.py` 有 SQLite 特化(WAL pragma、连接监听);async engine 绑 loop 的坑跨进程后形态变化,需专项验证。
- **worker 镜像体积/启动**:Chromium + playwright-mcp(Node)+ Python 同镜像,镜像大;case 级浏览器冷启动延迟要实测。
- **SSE 在多副本 API 后的会话黏性**:K8s ingress 需支持长连接;或改轮询兜底。
- **HTTP 工具覆盖不了的断言场景**(直查 DB):明确引导「项目方自建查询 HTTP 接口」,文档里写清楚边界。
- **LLM 网络可达性**:项目自带 LLM 的 api_base 必须从 K8s 集群内可达(内网代理/NO_PROXY 的老坑,部署文档要写)。
